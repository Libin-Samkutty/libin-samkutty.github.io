Python · DeepEval · RAGAS

AI Evaluation Pipeline

  Continuous evaluation in production

Problem: No structured evaluation infrastructure existed for production LLM outputs across 6 healthcare AI product lines. Hallucinations, prompt regressions, and retrieval quality degradation had no automated detection mechanism.

• Custom evaluation frameworks using DeepEval and RAGAS for answer relevancy, faithfulness, context precision and context recall, validated against curated golden datasets across 6 healthcare AI product lines serving 50,000+ monthly users across 6 countries.
• Automated production monitoring with trace instrumentation enabling continuous detection of prompt regressions, hallucinations, routing failures, intent misclassification and response quality degradation.
• End-to-end validation suites for semantic caching, RAG retrieval and LLM routing workflows, validating response consistency, retrieval accuracy, routing correctness and fallback inference behaviour.
• Risk-based regression suites covering multilingual conversational AI flows across 9 intent types and 17 routing labels, including edge cases and boundary conditions.
• Performance benchmarking and root-cause analysis across AI pipeline components: response time 12–18s → 4–7s, classification latency 2–3s → ~900ms, RAG retrieval 8–12s → 1.5–3.5s.

Result: Automated LLM evaluation and production monitoring across 6 healthcare AI product lines. Continuous hallucination and faithfulness detection in production. 60–70% improvement across pipeline response metrics through benchmarking and RCA.

Chatbot Testing Framework

Apache JMeter + OWASP ZAP

Performance and Security Test Suite

Problem: Performance baselines and security posture were validated manually and inconsistently - no sprint-level cadence, no documented baselines.

• API performance, load, and stress testing using Apache JMeter; baselines tracked across sprints to detect regressions.
• Per-sprint security scanning using OWASP ZAP; XSS injection vulnerabilities identified and reported to the development team.
• Reports delivered to the team at the close of each sprint cycle.

Result: XSS vulnerabilities surfaced and resolved within the sprint they were found. Performance baselines established a week-on-week view of API behaviour, making regressions visible across releases and informing sprint demos with concrete trend data.
